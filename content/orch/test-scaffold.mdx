# Test Scaffolding

Auto-detect your project stack and create the 5-layer TDD test directory structure in one command.

## Overview

`isa-orch test-scaffold` creates the standard isA test directory structure following the [TDD standard](/conventions#tdd-standard). It's idempotent вҖ” running it multiple times on a project that's already scaffolded is safe.

```bash
isa-orch test-scaffold [PATH]
```

`PATH` defaults to the current directory.

## Step 1 вҖ” Detect Project Context

Before scaffolding, understand your project's current state:

```bash
isa-orch project-context ./isA_Agent_SDK
```

```
Project context for: isA_Agent_SDK

Language:        Python 3.12
Framework:       FastAPI
Package manager: pip (pyproject.toml)
Test runner:     pytest
Test dirs found: tests/unit/, tests/component/
Missing layers:  integration, api, smoke
DB migrations:   Alembic (alembic/)
Docker:          Yes (Dockerfile)
CI:              GitHub Actions (.github/workflows/)
```

`project-context` reads:
- `package.json` / `pyproject.toml` / `setup.py` вҶ’ language and package manager
- `pytest.ini` / `jest.config.*` вҶ’ test runner
- `Dockerfile` / `.github/` вҶ’ infrastructure presence
- Existing `tests/` subdirectory names вҶ’ which layers already exist

## Generated Structure

```
tests/
в”ңв”Җв”Җ unit/           # L1: Pure functions, no I/O, no mocks needed
в”Ӯ   в””в”Җв”Җ .gitkeep
в”ңв”Җв”Җ component/      # L2: Classes/components with mocked dependencies
в”Ӯ   в””в”Җв”Җ .gitkeep
в”ңв”Җв”Җ integration/    # L3: Real DB, file I/O, multi-service interactions
в”Ӯ   в””в”Җв”Җ .gitkeep
в”ңв”Җв”Җ api/            # L4: HTTP endpoints, schema validation
в”Ӯ   в””в”Җв”Җ .gitkeep
в”ңв”Җв”Җ smoke/          # L5: Full user flows, end-to-end tests
в”Ӯ   в””в”Җв”Җ .gitkeep
в””в”Җв”Җ conftest.py     # Shared fixtures (empty template)
```

Generated `conftest.py` template:

```python
"""
Shared pytest fixtures for all test layers.

Conventions:
  - L1 (unit/): no fixtures needed вҖ” pure functions only
  - L2 (component/): mock external deps with pytest monkeypatch or MagicMock
  - L3 (integration/): use tmp_path or in-memory SQLite; avoid production DBs
  - L4 (api/): use test client (httpx.AsyncClient or FastAPI TestClient)
  - L5 (smoke/): full server running; may use subprocess or docker-compose
"""
import pytest
```

## TypeScript / React Projects

For TypeScript projects, scaffold creates Jest-compatible directories with placeholder config:

```
tests/
в”ңв”Җв”Җ unit/           # Jest unit tests
в”ңв”Җв”Җ component/      # React Testing Library component tests
в”ңв”Җв”Җ integration/    # Multi-component integration tests
в”ңв”Җв”Җ api/            # MSW (Mock Service Worker) API tests
в””в”Җв”Җ e2e/            # Playwright end-to-end tests
```

The `project-context` command detects `package.json` and sets up TypeScript-appropriate structure automatically.

## Integration with Skills

`isa-orch project-context` is called internally by:

| Skill | How it uses context |
|-------|-------------------|
| `/test` | Detects language вҶ’ selects pytest vs jest; detects existing layers вҶ’ skips creating duplicate tests |
| `/codegen` | Detects framework вҶ’ generates code matching FastAPI vs Flask vs Express patterns |
| `/run-tests` | Detects test runner вҶ’ uses `pytest` vs `npm test` |
| `isa-vibe` | Detects full stack вҶ’ routes CDD/TDD to correct guide |

After scaffolding, all skills work correctly without any manual configuration.
